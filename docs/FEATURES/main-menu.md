# 메인 메뉴 및 시스템 개요 (Main Menu & System Overview)

## Overview

반도체 시료 생산주문관리 시스템은 콘솔 기반으로 동작하며, 담당자가 프로그램을 실행하면 가장 먼저 마주치는 화면이 메인 메뉴(기능별 선택 화면)이다. 메인 메뉴는 전체 시료/주문에 대한 요약 정보를 함께 보여주고, 담당자의 번호 입력을 받아 각 기능(시료 관리, 시료 주문, 주문 승인/거절, 모니터링, 생산라인 조회, 출고 처리)으로 라우팅(dispatch)하는 진입점(entry point) 겸 네비게이션 허브 역할을 한다.

시스템 개요(전제 조건):

- 생산 라인은 **콘솔 기반**으로 동작하며, 담당자가 직접 명령(메뉴 번호)을 입력하는 **단일 프로세스** 구조이다.
- 생산 라인은 **단일 생산 라인**으로, 한 번에 하나의 시료만 생산할 수 있다 (동시에 여러 시료를 병렬 생산하지 않음). 이 제약은 생산라인 조회/모니터링 등 다른 기능 문서의 전제 조건이 되며, 이 문서에서는 시스템 개요로서만 언급한다.
- 화면 구성(문구, 레이아웃, 색상 등 UI 표현)은 자유롭게 결정 가능하다. 과제 스펙에 제시된 "예시 UI 화면"은 참고용일 뿐 강제 사항이 아니다. 이 문서가 규정하는 것은 **메뉴 구조와 라우팅 동작**이며, 화면 문구 자체는 범위 밖이다.

MVC 관점에서 메인 메뉴는:

- **Model**: 요약 정보(총 등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 건수)를 얻기 위해 다른 기능들의 Repository/Model을 조회(query)만 한다. 메인 메뉴 자체가 소유하는 별도 데이터 모델은 없다.
- **View**: 메뉴 목록과 요약 정보를 화면에 출력한다 (실제 콘솔 출력 포맷은 유닛 테스트 범위 밖).
- **Controller**: 사용자 입력(메뉴 번호)을 받아 유효성을 검사하고, 유효하면 해당 기능의 Controller를 호출(dispatch)하며, 무효하면 오류를 표시하고 재입력을 받는 라우팅 로직을 담당한다. 이 라우팅 로직이 이 문서의 핵심 테스트 대상이다.

## Menu structure

메인 메뉴에 표시되는 항목과, 각 항목이 연결되는 기능 문서는 다음과 같다.

| 번호 | 메뉴명 | 연결되는 기능 문서 |
|---|---|---|
| 1 | 시료 관리 (등록/조회/검색) | `docs/FEATURES/sample-management.md` |
| 2 | 시료 주문 (접수) | `docs/FEATURES/order-intake.md` |
| 3 | 주문 승인/거절 | `docs/FEATURES/order-approval.md` |
| 4 | 모니터링 | `docs/FEATURES/monitoring.md` |
| 5 | 생산라인 조회 | `docs/FEATURES/production-line.md` |
| 6 | 출고 처리 | `docs/FEATURES/shipment.md` |
| 0 | 종료 | 없음 (메인 루프 종료) |

각 메뉴 항목을 선택했을 때 실행되는 세부 동작(등록/조회/검색의 구체적인 흐름, 승인/거절 정책, 모니터링 집계 방식 등)은 이 문서의 범위가 아니며, 위 표에 연결된 각 기능 문서를 참조한다. 이 문서는 오직 **"어떤 번호가 어떤 기능으로 연결되는가"** 와 **"입력을 어떻게 처리하는가"** 만을 다룬다.

## Routing behavior — Acceptance Criteria

메뉴 라우팅(dispatch) 로직은 Controller 계층에서 테스트 가능하다. 테스트 시에는 콘솔 표준 입력을 직접 사용하지 않고, 가짜 입력 소스(fake input source, 예: 미리 정의된 입력 시퀀스를 반환하는 스텁/인터페이스)와 가짜 하위 컨트롤러(fake sub-controller, 예: 호출 여부만 기록하는 스파이/목)를 주입(inject)하여 검증한다.

**AC-1: 유효한 메뉴 번호(1) — 시료 관리 컨트롤러로 라우팅**

- Given 메인 메뉴 컨트롤러에 시료 관리(SampleManagement) 컨트롤러의 fake/spy가 주입되어 있고
- When 사용자가 메뉴 번호로 `1`을 입력하면
- Then 메인 메뉴 컨트롤러는 시료 관리 컨트롤러의 진입 동작(예: `run()`/`show()` 등)을 정확히 1회 호출하고
- And 다른 기능 컨트롤러(주문 접수, 승인/거절 등)는 호출하지 않는다.

**AC-2: 유효한 메뉴 번호(2~6) — 각 기능 컨트롤러로 라우팅**

- Given 메인 메뉴 컨트롤러에 6개 기능 컨트롤러 각각의 fake/spy가 주입되어 있을 때
- When 사용자가 `2`, `3`, `4`, `5`, `6` 각각을 입력하면
- Then 각 입력값에 대응하는 기능 컨트롤러(주문 접수, 주문 승인/거절, 모니터링, 생산라인 조회, 출고 처리)가 정확히 1회씩 호출된다 (표의 매핑과 일치).

**AC-3: 범위를 벗어난 숫자 입력 — 오류 표시 후 재입력**

- Given 메인 메뉴가 표시된 상태에서
- When 사용자가 정의되지 않은 번호(예: `7`, `99`, `-1`)를 입력하면
- Then 시스템은 크래시(비정상 종료)하지 않고
- And "유효하지 않은 메뉴 번호입니다" 취지의 명확한 오류 메시지를 표시하며
- And 어떤 기능 컨트롤러도 호출하지 않고
- And 다시 메뉴 번호 입력을 받는 상태(재프롬프트)로 돌아간다.

**AC-4: 숫자가 아닌 입력 — 오류 표시 후 재입력**

- Given 메인 메뉴가 표시된 상태에서
- When 사용자가 숫자가 아닌 값(예: `"abc"`, 빈 문자열, 공백)을 입력하면
- Then 시스템은 크래시하지 않고
- And 유효하지 않은 입력이라는 오류 메시지를 표시하며
- And 어떤 기능 컨트롤러도 호출하지 않고
- And 다시 메뉴 번호 입력을 받는 상태로 돌아간다.

**AC-5: 잘못된 입력 이후 유효한 입력 — 정상 복구**

- Given 사용자가 먼저 유효하지 않은 입력(`"abc"` 또는 `99`)을 입력해 오류를 겪은 상태에서
- When 이어서 유효한 메뉴 번호(예: `1`)를 입력하면
- Then 메인 메뉴 컨트롤러는 정상적으로 해당 기능 컨트롤러(시료 관리)를 1회 호출한다 (이전 오류가 이후 정상 라우팅에 영향을 주지 않는다).

**AC-6: `[0] 종료` — 메인 루프를 정상 종료**

- Given 메인 메뉴가 표시된 상태에서
- When 사용자가 `0`을 입력하면
- Then 메인 메뉴 컨트롤러는 어떤 기능 컨트롤러도 호출하지 않고
- And 메인 루프를 종료(exit)하며
- And 이후 추가 입력을 요구하지 않고 프로그램(또는 해당 세션 루프)이 정상적으로 종료 상태에 도달한다 (예외/크래시 없이).

**AC-7: 하위 기능 실행 후 메인 메뉴로 복귀**

- Given 사용자가 유효한 메뉴 번호(예: `1`)를 입력하여 하위 기능 컨트롤러 실행이 완료되었을 때
- When 하위 기능 컨트롤러가 정상적으로 반환(return)하면
- Then 메인 메뉴 컨트롤러는 다시 메인 메뉴를 표시하고 다음 메뉴 번호 입력을 받는 상태로 돌아간다 (`0`을 입력하기 전까지 루프는 계속된다).

## Summary info displayed on entry

메인 메뉴가 표시될 때(최초 진입 시, 그리고 각 하위 기능 실행 후 메인 메뉴로 복귀할 때), 다음 요약 정보를 함께 조회하여 표시한다. 이 값들은 각 기능의 Repository/Model을 **조회(read-only)만** 하여 얻으며, 메인 메뉴 자체가 새로운 계산/판단 로직(business logic)을 갖지 않는다.

| 요약 항목 | 설명 | 조회 대상 |
|---|---|---|
| 총 등록 시료 수 | 시스템에 등록된 전체 Sample 엔티티의 개수 | `docs/FEATURES/sample-management.md`의 Sample Repository |
| 총 재고 | 등록된 시료들의 재고 수량 합계 | `docs/FEATURES/sample-management.md`의 Sample 재고 필드 |
| 전체 주문 수 | 시스템에 접수된 전체 Order 엔티티의 개수 (상태 무관, 단 각 기능 문서에서 정의한 집계 제외 규칙이 있다면 그 규칙을 따름) | `docs/FEATURES/order-intake.md` / `docs/FEATURES/order-approval.md`의 Order Repository |
| 생산라인 대기 건수 | 현재 생산 대기 중(`RESERVED`→생산 전 등 생산라인 진입 대기 상태)인 주문 건수 | `docs/FEATURES/production-line.md` / `docs/FEATURES/monitoring.md`에서 정의한 대기 상태 기준 |

이 값들의 **정확한 산출 규칙**(예: `REJECTED` 주문 포함 여부, 재고 계산 방식 등)은 해당 소유 기능 문서에서 정의하며, 메인 메뉴는 이미 정의된 조회(query) 인터페이스를 호출해 값을 표시하기만 한다. 메인 메뉴 컨트롤러에 대한 테스트에서는, 각 조회 대상을 fake repository로 대체하여 "메인 메뉴가 조회된 값을 화면에 전달/포함하는지"만 검증하며, 집계 로직 자체(예: 재고 합산 공식)는 이 문서의 테스트 대상이 아니다 (해당 기능 문서의 테스트 대상).

## Testability note

- **테스트 가능한 부분 (unit test 대상)**: 메뉴 번호 입력 → 유효성 검사 → 올바른 하위 컨트롤러 dispatch, 무효 입력 처리 및 재프롬프트, `[0]` 종료, 요약 정보 조회 시 fake repository 호출 여부. 이는 가짜 입력 소스(fake input source)와 가짜 하위 컨트롤러(fake sub-controller/spy)를 Controller에 주입함으로써 GoogleTest로 검증 가능하다.
- **테스트 범위 밖 (out of scope for unit test)**: 실제 콘솔 표준 출력(stdout)의 문구, 정렬, 색상 등 화면 렌더링 자체. 이는 GoogleTest 유닛 테스트로 검증하지 않으며, 대신 `docs/FEATURES`의 수동 검증(manual verification) 절차를 통해 사람이 직접 확인하거나, 시간이 허락하는 경우 가벼운 골든 아웃풋(golden-output) 스모크 테스트(예: 특정 입력 시퀀스에 대한 stdout 전체를 캡처하여 스냅샷과 비교)로 보완할 수 있다. 골든 아웃풋 테스트는 필수가 아니며, 화면 문구가 자유롭게 바뀔 수 있다는 전제(Overview 참고)와 상충하지 않는 범위에서 선택적으로 적용한다.

## Non-Goals

이 문서는 메인 메뉴의 **네비게이션/구성(navigation & composition)**, 즉 "어떤 번호가 어떤 기능으로 연결되는가"와 "입력을 어떻게 처리/검증하고 하위 컨트롤러로 위임하는가"만을 다룬다. 다음 항목은 이 문서의 범위가 아니며 각 기능 문서에서 다룬다:

- 시료 등록/조회/검색의 세부 동작 및 검증 규칙: `docs/FEATURES/sample-management.md`
- 시료 주문 접수의 세부 동작(Order 생성, ID 채번 규칙 등): `docs/FEATURES/order-intake.md`
- 주문 승인/거절 정책 및 상태 전이: `docs/FEATURES/order-approval.md`
- 모니터링(현황 조회/집계) 로직: `docs/FEATURES/monitoring.md`
- 생산라인 조회 및 단일 생산 라인 운영 세부 규칙: `docs/FEATURES/production-line.md`
- 출고 처리 세부 동작: `docs/FEATURES/shipment.md`
- 요약 정보(총 등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 건수)의 산출 공식/집계 규칙: 각 값을 소유한 위 기능 문서
- 실제 콘솔 화면의 문구/레이아웃/색상 등 UI 표현: 자유 결정 사항이며 문서화 대상이 아님 (Testability note 참고)
